# Вопросы

1. Может ли пользователь быть в 2 командах? — Нет, т.к. в ручке /users/setIsActive в ответе User имеет одну команду (поле типа string). Соотношение User - Team — 1 ко многим. 
2. Делать ли отдельную таблицу для Team? — Да, в случае, если захотим добавить команде аттрибутов, это будет легко сделать
3. Просто баним пользователя или баним и переназначаем в ПР-е? 2 вариант, тк в PR не может быть неактивных пользаков
4. Надо ли назначать только что активированного пользователя на доступный ПР (например, если в команде есть ПР, на который назначено меньше 2 людей). — Да, потому что у нас нет ручки / возможности назначить ревьюера на конкретный ПР. ПР может остаться без ревьюера или только с одним ревьюером без возможности добавить нового.
5. Посокольку никаких требований по сортировке выходных данных не предъявлялось, она не произовдилась
6. ошибка в последней ручке reassign: в примере old_reviewer_id вместо old_user_id, как в схеме


# Конфигурация линтера
Форматеры:
1. goimports - автоматически организует импорты
2. gofumpt - строгий форматировщик кода

Линтеры
1. errcheck - проверяет обработку ошибок
2. govet - статический анализ, находит подозрительные конструкции
3. staticcheck - продвинутый статический анализ (все проверки кроме SA1019)
4. unused - находит неиспользуемый код
5. gocritic - предлагает улучшения стиля кода
6. gocyclo - проверяет цикломатическую сложность функций (порог: 15)
